# 仓库工作约定

这些指令定义本仓库的默认质量工作流。除非更具体的模块级指令覆盖，否则适用于 AI agent、自动化脚本和人工协作者。

## 沟通

- 说明保持简洁、围绕当前任务。
- 实现类任务需要报告决策、计划、变更文件、验证结果和残余风险。
- 需求含糊时，做最安全的合理假设并明确说明；只有当答案实质影响正确性、安全、数据或发布风险时才提问。
- 不要把猜测写成已验证事实。

## 事实来源优先级

当前仓库状态是事实来源。当代码、文档、issue、历史记录或记忆冲突时，按以下顺序处理：

1. 当前代码、配置、脚本、迁移、CI 配置和真实命令输出。
2. 当前文档、工程说明、运行手册、模块报告和契约快照。
3. 当前 issue、PR、release 记录和部署记录。
4. 仓库内维护的项目记忆、上下文索引或知识库。
5. 归档文档、旧计划、旧聊天上下文和未验证的口头约定。

如果低优先级信息与当前文件或命令输出冲突，必须先核验再行动；若过期信息属于本次交付范围，应同步修正。

## 核心质量规则

- 变更保持最小、可回滚，并严格贴合用户请求。
- 保护用户或协作者已有改动；除非明确要求，不回滚无关工作。
- 没有清晰必要性和验证证据，不新增生产依赖、不改变公共 API、不改迁移、不改权限、不改发布行为。
- 永不提交 secret、真实凭据、私钥、客户数据、生产密码或敏感个人信息。
- 反复出现的 review 规则应尽量沉淀成可执行检查。
- 重要验收标准都应能通过命令、日志、报告、截图、API 响应、CI 结果或 diff 证明。

## 风险分级

非平凡任务在编辑前先分级：

| 等级 | 典型任务 | 执行路径 |
| --- | --- | --- |
| 低风险 | 文档、文案、小样式、fixture、局部无行为变更 | 直接执行，跑最小验证 |
| 中风险 | 单模块实现，需求清晰 | 简短计划，运行目标测试或构建 |
| 高风险 | 跨模块、公共 API、认证、权限、安全、隐私、审计、事务、迁移、持久化、数据一致性、缓存、队列、并发 | 先计划，按小切片实现，独立 review 或等价证据 |
| 发布/事故风险 | 发布门禁、回滚、生产配置、生产数据、故障修复 | 先只读评估，写回滚方案，收集 runtime/release 证据 |

出现以下情况立即升级：

- 需求不明确，或需要猜测产品行为。
- 影响模块或文件数超过预期。
- 同一个测试或 gate 连续失败两次且原因不清。
- 涉及认证、权限、安全、隐私、审计、事务、迁移、持久化、数据一致性、缓存、队列、并发、发布或公共契约。
- 当前代码、文档和运行输出互相矛盾。

## 计划

非平凡任务必须维护实时计划。高风险或跨模块任务需要创建或更新可持久化计划，包含：

- 背景：issue/PR、触发原因、业务目标、当前事实。
- 非目标：本次明确不做什么。
- 风险等级：为什么这样分级，什么情况会升级。
- 可编辑范围：允许修改的模块/文件和禁止触碰的区域。
- 实施步骤：按小切片拆分，每步都有验证。
- 验收标准：每条都必须有证据。
- 验证矩阵：必跑 gate、条件 gate、跳过项。
- 回滚策略：代码、数据、配置、发布和文档如何回退。
- 完成同步：文档、issue、PR 说明、项目记忆或工作流是否需要更新。

计划必须与实际进度同步。工作范围扩大时，先停止宽泛编辑，重新定界。

## 协作

多人、多个工具或多个 agent 参与时，明确角色：

- Lead：负责事实、边界、决策、最终 review 和交付说明。
- Scout：只读发现文件、符号、文档、日志和命令输出。
- Implementer：只在分配范围内编辑。
- Reviewer：检查正确性、回归、安全、数据风险和缺失测试。
- Verifier：运行命令并记录证据。

规则：

- 一个任务只有一个最终负责人。
- 不递归委派。
- 同一文件同一时间只由一个实现者修改。
- 输出冲突时，以当前代码、可执行证据和任务目标裁决。
- 高风险变更在最终交付前需要独立 review 或等价书面审查。

## Harness

仓库应提供统一 harness 入口：

```bash
scripts/harness/check.sh <gate> [args]
```

建议 gate：

| Gate | 目的 | 默认强度 |
| --- | --- | --- |
| `quick` | 默认本地 PR 门禁，串联低环境依赖检查 | 大范围变更必跑 |
| `changed` | 根据变更文件选择检查 | 普通 PR 优先 |
| `docs` | 文档索引、链接、命名、修订记录、仓库卫生、空白检查 | 文档和工作流变更必跑 |
| `backend` | 后端编译、测试或质量门 | 后端变更必跑 |
| `frontend` | 前端类型检查、构建或测试 | 前端变更必跑 |
| `mobile` | 移动端静态审计、构建或 smoke | 移动端变更必跑 |
| `contract` | API、schema、客户端路径、公开枚举等契约检查 | 公共契约变更必跑 |
| `runtime` | 针对运行中服务采集 health、metrics、脱敏日志和可选 smoke | 显式诊断 |
| `e2e` | 依赖真实或临时运行环境的端到端业务闭环 | 显式或专项 CI |
| `release` | 发布前完整证据包 | 发布必跑 |
| `harness-tests` | harness 脚本和报告契约自测 | harness 变更必跑 |

Gate 边界：

- `quick` 不应依赖真实 secret、生产服务、不可控外部账号或手工工具。
- 环境重的 runtime 检查保持显式，除非它稳定、快速、可复现且失败易解释。
- 每个 gate 必须能单独运行。
- CI 应调用与本地一致的 harness 入口。
- gate 无法运行时，记录命令、原因、替代证据和后续补跑位置。

## Harness 报告

重要 gate 应支持：

```bash
HARNESS_REPORT_DIR=release-artifacts/harness/<gate> scripts/harness/check.sh <gate>
```

报告契约：

- `summary.json`：gate、状态、退出码、开始/结束时间、步骤列表。
- `summary.md`：面向人的摘要。
- `steps.jsonl`：每个步骤一条结构化记录。
- `logs/`：每个步骤的命令日志。

允许状态：`PASS`、`FAIL`、`SKIPPED`、`PLANNED`。跳过必须写原因和开启条件。报告和日志不得泄露 secret、token、密码、私钥、客户数据或敏感个人信息。

## `changed` 选择逻辑

`changed` 应降低验证成本，但不能降低质量。它应支持 plan-only 模式和覆盖变更文件列表，方便测试选择逻辑。

| 变更面 | 必选 gate | 条件 gate |
| --- | --- | --- |
| 文档 | `docs` | 空白检查 |
| 后端路由/控制器 | `backend`, `contract` | 前端/移动端构建、API smoke |
| DTO/schema/枚举 | `backend`, `contract` | 客户端 schema 报告 |
| 客户端 API wrapper | `contract`, `frontend` 或 `mobile` | runtime smoke |
| 前端页面/组件 | `frontend` | UI smoke 或截图 |
| 移动端页面/交互 | `mobile` | 设备/模拟器 smoke |
| 数据库/持久化 | `backend`, migration/schema check | runtime DB check, E2E |
| 认证/权限/审计/事务 | `backend`, guardrails, `contract` | API regression, 独立 review |
| Harness/CI | `harness-tests`, `quick` | 受影响专项 gate |
| 发布/部署配置 | `docs`, `release` 或 dry-run | runtime context, 回滚验证 |

## 文档 Gate

docs gate 应检查文档索引、active docs 本地链接、稳定文件名、文档性质、修订记录、阶段性文档标注、临时/备份/系统文件和 `git diff --check`。

当变更影响长期行为、验证口径、发布步骤、架构、backlog 状态或运行假设时，必须同步更新文档、运行手册、工作流文件或项目记忆。

## 契约 Gate

公共契约应在 PR 阶段失败，而不是运行时失败。优先覆盖：

- Route contract：HTTP method、path、handler、params。
- Static schema contract：request/response 类型、DTO 字段、enum 值。
- Client path contract：客户端 API 路径有服务端路由支撑。
- Public enum/catalog contract：客户端可见业务枚举来自显式来源。
- Runtime schema diff：显式诊断，除非稳定后再进入默认 gate。
- Client field alignment report：报告型诊断，等 ownership 和误报收敛后再升级。

只有有意变更契约时才更新快照。代码和快照变更必须在同一 PR。临时 allowlist 必须写原因、负责人和过期条件。

## 结构护栏

把高价值架构规则转成可执行测试或脚本。优先覆盖：

- API 响应不暴露 password、token、secret、privateKey 或敏感字段。
- JSON endpoint 使用统一响应 envelope；二进制、流、range/cache 响应作为显式例外。
- 需要 DTO/View 时，Controller 不直接返回持久化实体。
- 业务异常走集中处理。
- 审计、变更日志、outbox 或副作用写入有清晰服务边界。
- 成功审计和外部副作用在事务提交后执行，或有补偿机制。
- 写路径有明确事务边界，rollback-worthy 异常继续抛出。
- 服务端权限和数据范围检查不依赖隐藏 UI。
- 数据库约束尽量保护与服务层一致的业务 invariant。

短期 allowlist 可以接受，但必须有退出条件。

## Runtime、E2E 与 Release

Runtime gate 应采集健康状态、关键 metrics、脱敏错误日志尾部、可选 API regression 或 smoke 摘要，以及必要的 runtime schema、数据库约束或迁移状态。

Release gate 应记录 commit、分支、目标环境、参数、配置检查、数据库检查、构建/测试、显式开启的重型 E2E/smoke、跳过原因、报告路径和失败日志。

服务镜像和运行依赖版本应精确固定。不要对 release-critical 依赖使用浮动 `latest` 标签。

## 安全

- 永不提交真实 secret 或客户数据。
- 示例配置必须使用明显占位值。
- runtime 检查应发现缺失、空值、占位值和不安全默认值。
- 日志、artifact 和上传报告必须脱敏。
- 认证、权限、隐私、审计、事务、持久化、迁移、数据一致性和发布变更默认属于高风险。

## 完成标准

最终交付前：

- 用户请求的行为已实现，且没有混入非目标。
- 相关文档、运行手册、索引、issue、PR 说明或项目记忆已更新，或确认无需更新。
- 已通过最窄但有意义的验证。
- 高风险变更已有专项 gate、runtime/release 证据或独立 review。
- 失败或跳过的验证列明原因和后续动作。
- 没有覆盖用户或协作者未授权改动。
- 最终回复包含变更文件、验证结果、残余风险和必要的后续 review 建议。
